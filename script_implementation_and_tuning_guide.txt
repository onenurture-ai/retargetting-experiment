# Replay Allegro Grasp Isaac Sim - Implementation Summary and Tuning Guide

This document summarizes the implementation details of the `replay_allegro_grasp_isaacsim.py` script,
including the key files involved, core algorithms, and parameter tuning recommendations for improving
retargeting quality.

## 1. Feature Overview

The script implements the following functionality in the NVIDIA Isaac Sim simulation environment:
1.  **Allegro Hand Grasping Simulation**: Loads the Allegro dexterous hand and smoothly moves it from an initial pose (all zeros) to a target grasp pose (grasp_1).
2.  **Real-time Retargeting**: Maps the Allegro Hand's motion in real time to the Shadow Hand, making the Shadow Hand mimic the Allegro's movements.
3.  **Data Recording**: Records the world-coordinate fingertip trajectories of both hands during simulation and saves them as a CSV file.
4.  **Time Control**: Controls the simulation speed for the first 120 frames so that they complete in approximately 6 seconds (~20 FPS), making it easier to observe and synchronize.

## 2. Detailed Implementation Steps

The script's execution flow is divided into the following stages:

### 2.1 Initialization Phase
*   **Start Isaac Sim**: Create a `SimulationApp`.
*   **Scene construction**: Create `/World`, add a ground plane, lighting, a cube, and other basic objects.
*   **Load robots**:
    *   **Allegro Hand**: Load the USD file, configured with PD control (Position Control) to execute the grasping action.
    *   **Shadow Hand**: Load the USD/URDF file, configured with **pure kinematic control** (Kinematic Control, stiffness/damping set to 0) to follow the retargeting result.
*   **Load retargeter**:
    *   Read the `shadow_hand_right.yml` configuration file.
    *   Initialize the `dex_retargeting` `SeqRetargeting` object, building the optimizer for the Shadow Hand.

### 2.2 Main Loop Phase
At each simulation frame, the script executes the following logic:

1.  **Time control and interpolation (frames 0–120)**:
    *   Use `time.perf_counter()` to monitor elapsed time.
    *   If the frame count is <= 120, compute the expected elapsed time for the current frame; if running too fast, call `time.sleep`.
    *   **Linear interpolation**: Compute the current target joint angles `current_target` for the Allegro hand (linearly transitioning from `initial_q` to `target_q`) and set them on the Allegro's drive.

2.  **Physics stepping**:
    *   Call `world.step(render=True)` to advance the physics simulation.

3.  **Allegro Forward Kinematics (FK)**:
    *   Read the current real joint angles of the Allegro hand from Isaac Sim.
    *   Convert the joint angles to the order required by the Pinocchio library.
    *   Use `RobotWrapper` (based on Pinocchio) to compute the positions of Allegro's **8 keypoints** (4 fingertips + 4 middle segments) in the Allegro base frame.

4.  **Coordinate frame transform and mapping**:
    *   **Rotation alignment**: Rotate the keypoints from the Allegro frame by -90 degrees (about the Z axis) to align with the Shadow Hand's coordinate frame (compensating for the difference between Allegro's palm-forward-Z and Shadow's palm-forward-Y orientation).
    *   **Fingertip mapping**: Map the 8 Allegro keypoints to the 10 Shadow target points (`ref_value`).
        *   *Note*: Since Allegro has only 4 fingers, the script maps Allegro's ring finger data to both Shadow's ring finger and little finger.

5.  **Retargeting solve**:
    *   Call `shadow_retargeting.retarget(ref_value)`.
    *   The optimizer computes the Shadow joint angles `q_shadow` that bring the Shadow fingertips closest to the target points `ref_value`.

6.  **Drive Shadow**:
    *   Map the computed `q_shadow` back to Isaac Sim's joint order.
    *   Call `shadow_art_view.set_joint_positions(...)` to directly set the Shadow hand's joint angles, teleporting it to the solved pose.

7.  **Data recording**:
    *   Compute the world coordinates of all fingertips for both hands and write them to `fingertip_coords.csv`.

## 3. Core Files and Libraries Involved

### 3.1 Scripts and Configs
*   **Script**: `replay_allegro_grasp_isaacsim.py` (main program)
*   **Shadow config**: `src/dex_retargeting/configs/offline/shadow_hand_right.yml`
    *   Defines the retargeting type (Position), URDF path, target keypoints (Target Links), etc.
*   **Allegro config**: `src/dex_retargeting/configs/offline/allegro_hand_right.yml`
    *   Primarily used to retrieve Allegro's URDF path and keypoint names.

### 3.2 Robot Models
*   **Allegro URDF**: `assets/robots/robots/hands/allegro_hand/allegro_hand_right.urdf`
*   **Shadow URDF**: `assets/robots/robots/hands/shadow_hand/shadow_hand_right.urdf`

### 3.3 Core Library (`dex_retargeting`)
*   **`seq_retarget.py`**: Implements the `SeqRetargeting` class, handles time-series retargeting, and includes low-pass filtering for smooth output.
*   **`optimizer.py`**: Implements the `PositionOptimizer` class, solving the nonlinear optimization problem (Inverse Kinematics) using NLOPT or SciPy.
*   **`robot_wrapper.py`**: Wraps the Pinocchio library to compute forward kinematics (FK) and Jacobian matrices.

## 4. Algorithms Used

1.  **Optimization-based Retargeting**:
    *   **Type**: Position-based Retargeting.
    *   **Objective function**: Minimize the Euclidean distance between Shadow keypoint positions and the target positions (from Allegro).
    *   $$ \min_{q} \sum_{i} w_i \| P_{shadow, i}(q) - P_{target, i} \|^2 + \text{regularization} $$
    *   **Constraints**: Respect the Shadow Hand's joint angle limits (Joint Limits).

2.  **Forward Kinematics**:
    *   Uses the **Pinocchio** rigid-body dynamics library to compute the Cartesian-space positions corresponding to given joint angles.

3.  **Time-series smoothing**:
    *   `SeqRetargeting` uses the previous frame's solution as the initial guess for the current frame (Warm Start), ensuring solution continuity and avoiding abrupt pose changes.

## 5. Tuning Recommendations: How to Improve Mapping Quality

If the Shadow Hand's motion following is unsatisfactory, try the following adjustments:

### 5.1 Coordinate Frame and Scale Alignment (Most Critical)
*   **Rotation matrix (`R_allegro_to_shadow`)**:
    *   The script currently uses a fixed rotation matrix (rotate -90 degrees about Z). If the palm orientations are inconsistent, this can cause incorrect finger bending directions.
    *   *How to adjust*: After `transform_points_to_world` or `compute_forward_kinematics`, inspect and modify the rotation matrix in the script.
*   **Scale factor (Scaling)**:
    *   The Allegro and Shadow hands have different palm sizes. If Shadow's fingers consistently cannot reach the target points, or bend excessively, the target points may be too far or too close.
    *   *How to adjust*: Introduce a scaling factor when computing `ref_value` in the script.
    *   Example: `ref_value[i] = allegro_in_shadow_frame[i] * 1.1` (expand target range) or `* 0.9` (shrink).

### 5.2 Keypoint Mapping Logic
*   **Finger correspondence**:
    *   The current script maps Allegro's Ring Finger to both Shadow's Ring and Little (Pinky) fingers.
    *   *How to adjust*: If you do not want the little finger to follow the ring finger, modify the `ref_value` assignment logic, e.g. keep the little finger at a fixed position or apply an offset based on the ring finger position.
*   **Keypoint selection**:
    *   Currently using fingertips (Tip) and middle segments (Middle).
    *   *How to adjust*: If finger bending pose is incorrect, add more keypoints (e.g. proximal joint positions) in `shadow_hand_right.yml` and the script to better constrain the full finger shape.

### 5.3 Config File Adjustments (`shadow_hand_right.yml`)
*   **Low-pass filter (`low_pass_alpha`)**:
    *   Current value is `1.0` (no filtering).
    *   *How to adjust*: If Shadow's motion is jittery, reduce this value (e.g. `0.2` to `0.5`). Note that this increases latency.
*   **Joint limits**:
    *   The optimizer strictly respects the joint limits defined in the URDF.
    *   *How to adjust*: If Shadow cannot achieve certain poses, check the `limit` attributes in `shadow_hand_right.urdf` and loosen the range (`lower` and `upper`) for specific joints as needed.

### 5.4 Optimizer Weights
*   `dex_retargeting` allows assigning weights to different keypoints.
*   *How to adjust*: If you want the thumb to match more accurately while tolerating more error on the little finger, increase the weight for thumb keypoints during optimizer initialization (requires modifying the `optimizer` init parameters).

### 5.5 Initial Pose
*   If Shadow starts in a very unusual pose, the optimizer may converge to a local optimum.
*   *How to adjust*: Adjust the Shadow hand's initial joint angles in Isaac Sim to a pose closer to the Allegro hand's initial state.
